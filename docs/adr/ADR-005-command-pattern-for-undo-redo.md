# ADR-005: 使用命令模式实现撤销/重做

## 状态
已接受

## 日期
2025-12（项目启动时）

## 背景
座位安排过程中，用户需要：

- 撤销意外操作（如错误地交换了两个学生的座位）
- 重做已撤销的操作
- 支持多步撤销（不仅限于一步）
- 操作包括：自动策略执行、手动拖拽交换、单个座位分配/取消分配
- 撤销/重做后，UI 状态需要正确恢复（绑定属性、选择状态等）

## 决策
使用**命令模式**实现 `IUndoableCommand` + `CommandHistory`：

- `IUndoableCommand` 接口定义 `ExecuteAsync` 和 `UndoAsync`，均接收 `SeatingWorkspace` 作为操作上下文
- `CommandHistory` 维护两个栈：`_undoStack` 和 `_redoStack`
- 执行新命令时，推入撤销栈并清空重做栈（新操作使重做历史无效）
- 撤销时，从撤销栈弹出，执行 `UndoAsync`，推入重做栈
- 重做时，从重做栈弹出，执行 `ExecuteAsync`，推入撤销栈

## 考虑的替代方案

### Memento 模式（完整状态快照）
- 优点：实现简单，无需为每个操作编写 `Execute`/`Undo` 对
- 缺点：每次操作保存完整座位状态快照，内存开销大（数百个座位 × 多次操作）；快照之间难以表示"增量"
- 拒绝：内存效率差。命令模式记录操作意图而非完整状态，内存友好

### 事件溯源
- 优点：完整审计日志，可从任意时间点重建状态
- 缺点：复杂度高；座位安排的每个修改都需要序列化为事件；重放性能不确定
- 拒绝：项目已通过 `SeatingSnapshot` 提供历史版本管理，不需要事件溯源级别的粒度

### 不实现撤销（依赖"重置"按钮）
- 优点：零开发成本
- 缺点：用户体验差 — 一步错误需要全部重新安排
- 拒绝：座位安排操作复杂，没有撤销会严重影响用户信心

## 后果
- 每个可撤销操作（策略执行、手动拖拽、座位分配）必须实现为 `IUndoableCommand`
- 撤销/重做栈通过 `IApplicationFacade` 暴露给 UI 层（`ExecuteCommandAsync`、`UndoAsync`、`RedoAsync`）
- `SeatingWorkspace` 的变更日志 (`_changeLog`) 记录每次修改的增量，命令使用这些记录来逆向操作
- 命令栈只存储增量变化（"座位 A 分配给学生 X"），而非完整快照，内存效率高
- `CanUndo` / `CanRedo` 属性绑定到 UI 按钮的 `IsEnabled` 状态
